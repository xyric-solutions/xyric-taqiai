## Why this is one buildable unit
Account lifecycle (register → login → session → profile) is a single cross-cutting foundation: every user-scoped feature depends on it, and it is verifiable end-to-end in one pass by registering, authenticating, and reading the JWT-protected session.

## Acceptance (definition of done for the segment)
- [ ] A new user can register with email, password, and Pakistani lawyer fields (barCouncilId, phone, city, language) and receive a session.
- [ ] Login issues a signed JWT (jose) set as an httpOnly cookie; passwords are bcrypt-hashed, never stored plaintext.
- [ ] `middleware.ts` rejects unauthenticated requests to protected routes and passes the decoded user to handlers.
- [ ] Profile read/update and change-password work against the authenticated user only (no cross-user access).
- [ ] Test expectation: integration tests cover register/login happy path, wrong-password rejection, expired/invalid JWT rejection, and change-password re-auth.
- [ ] NFR checks: passwords bcrypt-hashed; JWT httpOnly + signed; user-scoped queries enforce ownership.
- [ ] Verification: `POST /api/auth/register` then `GET /api/auth/session` with the returned cookie → 200 with the user's profile.
