## Why this is one buildable unit
This is a single narrow assist endpoint plus its suggestion logic: it takes the field currently being filled and returns a realistic hint, so it is buildable and verifiable in one pass by requesting a suggestion for a given field and asserting identity fields return format-only.

## Acceptance (definition of done for the segment)
- [ ] For a drafting-form field, the API returns a realistic, Pakistan-appropriate suggested value that the lawyer can accept or override (hint, not a binding fact).
- [ ] Identity fields (CNIC, exact names, account numbers) return placeholder/format guidance only — never a fabricated concrete value.
- [ ] Suggestions never overwrite user input silently; they are surfaced as suggestions in the form.
- [ ] Test expectation: unit tests assert non-identity fields get concrete hints and identity fields (CNIC/name/account) get format/placeholder only.
- [ ] NFR checks: no fabricated identity data; suggestions are advisory (MoSCoW: Could).
- [ ] Verification: `POST /api/ai/suggest` for a CNIC field → format placeholder (e.g. `#####-#######-#`), not a fabricated number.
