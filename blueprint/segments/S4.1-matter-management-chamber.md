## Why this is one buildable unit
Every matter operation — create, view, update, archive, and link documents/hearings — flows through one canonical Matter model and its `/api/matters` surface behind the `/chamber` UI, so the whole practice-management core is a single buildable, verifiable unit.

## Acceptance (definition of done for the segment)
- [ ] `/chamber` is the canonical Matter manager; `/cases` and `/cases/[id]` render the same Matter data and redirect legacy paths.
- [ ] Full CRUD works with 100% reliability: create, list, get, update, and archive a Matter without data loss.
- [ ] A Matter stores client (name/CNIC/phone), court, case number, role (advocate/client), opponent, judge, and dates; `documentIds` JSON preserves document-link integrity across updates.
- [ ] `MatterHearing` rows (date/purpose/result/nextDate) attach to a Matter and drive the next-hearing value.
- [ ] Legacy `/api/cases/*` retained only as a redirect to `/api/matters`; the retired `LegalCase`/`cases` model is NOT extended.
- [ ] Test expectation: integration tests on `/api/matters` CRUD + `/api/matters/[id]/hearings`, asserting document-link integrity and legacy-redirect behaviour.
- [ ] NFR checks: CRUD reliability 100%; document-link integrity maintained; no writes to retired LegalCase model.
- [ ] Verification: `POST /api/matters` then `GET /api/matters/[id]` → returns the created Matter with intact `documentIds`; `GET /api/cases/[id]` → redirects to the Matter route.
