## Why this is one buildable unit
The scorecard store, its metrics, the per-module QA gates, and the results tracker are one closed measurement loop (LEGAL-03 Validation Mode) that only makes sense built together; it is never lawyer-facing and gates every other AI module's quality.

## Acceptance (definition of done for the segment)
- [ ] Every core AI module (Drafting, Advisor, OCR, Voice, Judgment) has at least one scorecard entry from a measured run.
- [ ] Scorecard metrics computed and stored: citation accuracy, argument completeness, structural compliance, hallucination rate — with targets tracked over time (trend).
- [ ] Validation surface is admin-only; no lawyer-facing route exposes scorecards or filed-document corpus.
- [ ] Test expectation: unit tests for each metric scorer; integration test running one `ValidationRun` against a fixture case → persisted scorecard; regression baseline recorded.
- [ ] NFR checks: testing corpus (actual filed docs of solved cases) access-controlled; PII in the corpus not leaked into logs or client responses.
- [ ] Verification: run a `ValidationRun` for the Drafting module against a known solved case → scorecard rows written and `testing/Results-Tracker.md` updated with the measured numbers.
