## Why this is one buildable unit
The statute corpus, its FTS5 index, the search API, and the statute-search page form one grounding unit: a lawyer looks up a section and gets the actual latest Act text with status and province — verifiable in one pass (query section → real provision text → status/province badges), and it feeds the Advisor's RAG grounding.

## Acceptance (definition of done for the segment)
- [ ] `/statute-search` resolves a section query (e.g. PPC/CrPC/CPC/MFLO/SRA + section no.) to the real provision text, never a fabricated section.
- [ ] `/api/statutes/search` runs FTS5 over `sections_fts` in read-only `data/statutes.db` and returns act, section, status, and province.
- [ ] `retrieveStatuteGrounding` supplies grounded statute snippets for downstream RAG (Advisor, S2.5).
- [ ] Corpus spans Federal + Punjab/Sindh/KPK/Balochistan; `LegalAct.docType` (act/regulation/ordinance) and status (in-force/amended/superseded) are surfaced.
- [ ] Test expectation: integration tests assert real-section retrieval, "not found" for a non-existent section (no invention), status/province surfacing, and FTS5 rebuild via `build_statute_fts.py`.
- [ ] NFR checks: zero invented sections; statute text read-only; status/province always shown.
- [ ] Verification: `GET /api/statutes/search?q=<act+section>` → real provision text + status + province; a non-existent section → empty/not-found, never fabricated.
