## Why this is one buildable unit
Semantic search is a single retrieval unit: a sidecar embedding service plus the Next.js API that calls it and silently degrades to keyword search when the service is down — verifiable in one pass by querying with the service up (vector hits) and with it stopped (keyword hits, no user-facing error).

## Acceptance (definition of done for the segment)
- [ ] `scripts/semantic_service.py` runs on port 8137, loads fastembed model `paraphrase-multilingual-MiniLM-L12-v2` over `data/semantic.db`, and serves `/health`, `/search?q=&k=`, `/reload`.
- [ ] `/api/judgments/semantic` returns meaning-ranked judgments when the service is healthy.
- [ ] When the service is unreachable, `/api/judgments/semantic` transparently falls back to `/api/judgments/local` keyword search — ZERO user-facing failures.
- [ ] `/reload` refreshes the vector store after corpus growth without a service restart.
- [ ] Test expectation: integration tests assert vector results when up, keyword-fallback results (200, no error) when the service is stopped, and `/health` gating.
- [ ] NFR checks: graceful degradation (fallback on service-down); multilingual (Urdu/English) query support.
- [ ] Verification: stop the service → `GET /api/judgments/semantic?q=<term>` still returns 200 keyword results; start it → same query returns vector-ranked results.
