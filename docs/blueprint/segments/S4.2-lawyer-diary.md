## Why this is one buildable unit
The diary is one page over one `DiaryEntry` model and its `/api/diary` CRUD surface, so recording a hearing and reading back its next date is a single buildable, verifiable unit.

## Acceptance (definition of done for the segment)
- [ ] `/lawyer-diary` lists diary entries and supports create, edit, and delete.
- [ ] Each entry stores case number, court name, stage, proceeding, last date, next date, and client phone.
- [ ] Next-hearing accuracy is 100%: the `nextDate` stored is the value returned and surfaced to the dashboard cause list.
- [ ] `/api/diary` (list/create) and `/api/diary/[id]` (get/update/delete) are the only write surface for diary entries.
- [ ] Test expectation: integration tests on `/api/diary` and `/api/diary/[id]` covering create/update/delete and next-date round-trip fidelity.
- [ ] NFR checks: next-hearing accuracy 100%; dates persisted without timezone drift.
- [ ] Verification: `POST /api/diary` with a `nextDate` then `GET /api/diary/[id]` → returns the same `nextDate` and entry fields unchanged.
