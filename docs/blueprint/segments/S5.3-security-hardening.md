## Acceptance (definition of done for the segment)
- [ ] No default/fallback secret exists anywhere in the codebase; app fails fast at startup if `JWT_SECRET` is missing.
- [ ] Rate limiting is persistent and cross-instance (Redis/Upstash or Postgres), keyed by user-ID + IP, with stricter caps on OCR / voice / judgment / generate.
- [ ] Responses carry CSP, HSTS, and Permissions-Policy headers; no provider/internal AI error text reaches clients.
- [ ] All AI/mutation routes validate input via shared `zod` schemas with max prompt length and file type/size limits.
- [ ] Test expectation: unit tests for the limiter and error normaliser; integration tests asserting 401 when secret unset (dev harness), 429 after cap, and generic error body on provider failure; header presence asserted in E2E.
- [ ] NFR checks: production JWT secret rotated; secrets never logged; headers scored on securityheaders.com equivalent.
- [ ] Verification: unset `JWT_SECRET` locally → server refuses to boot; `curl -I` on a route → CSP/HSTS/Permissions-Policy present.
